# AgentEval-Lite 项目选址决策记录

> 日期：2026-07-06
> 结论口径：本文件记录 AgentEval-Lite 为什么作为独立开源项目维护，而不是嵌入某个具体业务 Agent 仓库。

## 1. 结论

AgentEval-Lite 应作为独立 Java 17 + Maven 项目维护，产物是单 jar CLI 与目录化任务库。

不把评估框架放进业务 Agent 仓库，核心原因是评估框架的生命周期、依赖边界和安全职责都不同：业务 Agent 是被评对象，AgentEval-Lite 是评估工具与裁判。二者仓库隔离，能避免发布节奏、依赖选择、权限模型和测试门禁互相纠缠。

## 2. 设计约束

- 评估对象应是任意 Agent：manual、scripted、cli、http、docker sandbox，不绑定某个应用框架。
- 评估任务应目录化：`task.yaml`、`work/`、`hidden/`、`samples/` 可整体版本化、审查和复用。
- 判分必须可复现：提交、workspace、hidden、trace 与 engine 版本都要进入报告指纹。
- 评估框架不应引入业务运行时依赖：不需要 Spring、数据库或线上服务进程。
- 安全边界必须清晰：`hidden/` 是评审禁区；不可信 Agent 用 Docker 沙箱只挂载 Agent 可触碰区。

## 3. 技术选型

| 维度 | 选择 | 理由 |
| --- | --- | --- |
| 运行时 | Java 17 | 易分发、易在 CI 中运行，适合单 jar CLI |
| 构建 | Maven | 与 Java 生态常规工具链一致 |
| CLI | picocli | 轻量、无服务框架依赖 |
| JSON/YAML | Jackson | 任务规格、提交、报告、trace 都依赖结构化数据 |
| Schema | networknt json-schema-validator | 提交契约需要 JSON Schema fail-fast 校验 |
| 测试 | JUnit 5 + AssertJ | 单元、集成和端到端回放覆盖 |

## 4. 独立成项目的理由

1. **裁判与选手分离**：业务 Agent 不应携带自己的隐藏评审规则，也不应控制评估报告生成链路。
2. **依赖边界更小**：评估 CLI 不需要业务应用的 Web 框架、存储、鉴权或部署依赖。
3. **任务资产可复用**：内置 `tasks/` 是框架自测库，用户私有测评集可放在 `evalsets/<set>/`，两者都不依赖具体业务仓。
4. **安全模型更清楚**：Work/Judge 隔离、trace 签名、tool allowlist、Docker sandbox 都由评估框架统一实现和回归。
5. **CI 门禁更稳定**：框架改动走 `mvn test`、`suite`、`redteam`、CODEMAP 漂移门禁；业务系统不需要承担这些工具自身的回归成本。

## 5. 当前仓库边界

本仓库负责：

- CLI 命令面与 Java 核心实现
- JSON Schema 与任务规格加载校验
- 内置示例任务与 replay 样例
- hidden judge、trace、report、suite、history、export
- 红队回归与 CI smoke 门禁
- 面向贡献者和 AI 编码助手的协作指南

本仓库不负责：

- 托管模型服务
- 业务 Agent 的线上运行时
- 企业权限系统
- 持久化结果平台或可视化 dashboard
- 真实业务数据集

## 6. 开源注意事项

- 示例任务和 demo evalset 必须只使用合成或脱敏数据。
- `runs/`、`target/`、真实凭证、本地环境文件不入库。
- 公开文档不写本机绝对路径、私有仓库结构、真实客户信息或隐藏答案。
