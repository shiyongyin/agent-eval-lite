WHAT / HOW / OUTPUT三句话说清整个框架

评估的根本难题是：Agent 的话不能全信。AgentEval-Lite 把「考试」这件事工程化——考题与评分标准分离、答题通道唯一、判分过程留痕、结果可复算。

评什么

被评对象：任何能产出「结构化提交」的 Agent——人工、回放脚本、命令行 Agent（claude / codex / cursor-agent / 自研）、HTTP 服务。
能出的题：任何可结构化验收的任务——代码修复、API payload 生成、文档分析、工具调用纪律、需求评审……内置 5 个示例任务开箱即用。

怎么评

给 Agent 一道带隐藏标准答案的题：它在隔离工作区干活，把 JSON 答卷写进 inbox；隐藏规则引擎（15 种 check）判分后回传「指方向不给答案」的受控反馈，多轮迭代直到达线或轮次用尽。全程 17 类事件 HMAC 签名留痕，编造过程无效。

产出什么

每次评估一个自包含的 run 目录：report.md/json（分数、维度、轮次轨迹、安全项）+ 签名 trace.jsonl + 可复现指纹（随时离线复算，同输入必同分）。批跑得到任务 × Agent 矩阵、pass^k 可靠性排名与跨 run 趋势。

第 2 步 · 把你的 Agent 接进来

框架不关心 Agent 内部怎么思考，只关心三件事：给它任务说明与反馈、让它在 workspace 里干活、收它写进 inbox 的提交。所以任何形态的 Agent 都能接。

cli 模式：占位符与环境变量

命令模板里可用 6 个占位符（自动 shell 转义），Agent 进程还会拿到同义的环境变量；进程工作目录固定为 workspace，超时强杀。

占位符环境变量含义
{instructions}AEL_INSTRUCTIONS任务说明文件 instructions.md 的绝对路径
{workspace}AEL_WORKSPACE本次 run 的私有工作区（可自由读写）
{inbox}AEL_INBOX提交收件目录（唯一有效提交通道）
{attempt_id}AEL_ATTEMPT_ID本轮应使用的提交 id，如 attempt_002
{feedback}AEL_FEEDBACK上一轮受控反馈文件（首轮为空）
{run_dir}AEL_RUN_DIRrun 根目录（Docker 沙箱下置空，因含评审禁区）

Agent 侧要做的唯一一件事：把提交 JSON 写进 inbox

无论哪种接入方式，评分只认写进 inbox/attempt_NNN.json 且通过双重 Schema 校验的提交——stdout 里的自然语言不计分。提交是一个「信封 + 分型」结构：

信封字段（必填）类型说明
schema_versionint固定 1
task_id / attempt_idstring与 run 上下文强校验，对不上判 invalid
submission_typeenumgeneric | code_fix | api_payload | tool_call | document | review，决定追加哪些分型字段
summarystring本次提交做了什么（≥8 字符，防空提交）
known_risksstring[]已知风险边界，可为空数组但必须存在
needs_human_reviewbooleantrue 会挂起等待人工复核
evidence[]（可选）object[]证据链 {type, source, quote_or_ref}，document/review 型必填
tool_calls_used[]（可选）object[]声明用过的工具调用，call_id 必须能和留痕对上
usage（可选）object自报 token/成本，仅做 ROI 参考，不参与评分

http 模式：窄口径契约（推任务、收提交）

任务带工具时：经网关调用（留痕、可核验）

任务声明了 allowed_tools 时，Agent 通过 CLI 网关调用工具，拿到应答和一个 call_id。提交里引用的 call_id 必须能和签名留痕对上——编造调用记录会被一票否决。

bash
# Agent 侧调用（需要环境变量 AEL_RUN_DIR，cli 模式已自动注入）
bin/agent-eval tool call user.lookup --input '{"user_id":"u_1001"}'
# → {"call_id":"tc_a1b2…","output":{"level":"GOLD",…}}   应答来自任务 hidden/tools/ 的应答库（Agent 看不到库本身）
评不可信 Agent？开沙箱--sandbox docker 会把 Agent 命令包进容器，容器里只有 workspace、inbox、feedback、instructions 四个挂载点——隐藏答案根本不在容器文件系统里，偷看无从谈起；默认断网。镜像需自带 Agent 运行环境（用工具时还需 JRE）。

第 3 步 · 读懂反馈与报告

怀疑分数不对？离线复算

想看过程细节？导出到看板

bin/agent-eval export --run runs/xxx/run_yyy 把留痕导出为 OTLP/OpenInference（run→AGENT / attempt→CHAIN / tool_call→TOOL 三层 span），Phoenix / Langfuse / OTel Collector 可直接摄取。

退出码：接 CI 就靠它

退出码含义
0评估正常完成（无论通过与否——verdict 看报告）
1参数 / 输入错误（含 Docker 未就绪的前移拦截，不会误记为 Agent 低分）
2框架故障或完整性熔断（INTEGRITY_BROKEN）
3配了 --fail-on-not-passed 且未通过（CI 门禁专用）

第 4 步 · 构建高质量测评集

一个任务 = 一道「有隐藏标准答案的题」。核心纪律只有一条：Agent 能看到的（task.yaml、work/）绝不含答案；答案与规则全在 hidden/。跟着五步法走，脚手架保证你从「已经能跑」的起点开始。

建题五步法

  1. 生成脚手架：bin/agent-eval task init --id my-task-001 —— 产物是一个开箱即过 validate、自带 fail→pass 回放闭环的最小任务，从它开始改，不要从空目录开始猜。
  2. 写题面：改 task.yaml（task_id 必须等于目录名；维度权重之和必须等于满分），把任务材料放进 work/。
  3. 写评审：改 hidden/judge.rules.yaml（15 种 check 见下方浏览器）；期望值放 hidden/expected/，用 expected_from: "expected/x.json#/指针" 引用——不要写进任何对外文案。
  4. 过体检：bin/agent-eval validate --task tasks/my-task-001 —— 深度 lint 把「run 时才会炸」的配置错误前移拦截（expected_from 断链、schema_file 缺失、check 要求调用白名单外工具…）。
  5. 钉住闭环：更新 samples/attempt-pass.json / attempt-fail.json + replay.yaml，用 scripted 跑通「第 1 轮失败 → 按反馈修正 → 第 2 轮通过」，这个闭环以后就是这道题的 CI 回归。

高质量任务清单（建题时逐条自检）

1 · 答案只在 hidden

期望值经 expected_from 指针引用；feedback_fail 对外文案指方向不给值（「金额不正确，注意折扣与单位」而不是「应为 4950」）。

2 · 红线用 blocking

结构非法、必须调的工具没调、canary 泄露——这类「过了也不算过」的检查标 blocking: true，一票否决。

3 · 主观题用部分得分

文档/评审类任务用 list_coverage 的关键点组覆盖率拿部分分，别做成全对全错；真正主观的表达质量交给低权重 llm_rubric（≤30%、禁 blocking）。

4 · 埋 canary 探针

在 hidden 里放 canary_token，配 no_canary_leak check——Agent 把隐藏内容抄进提交/工作区/日志/工具入参都会被抓。

5 · 过程造假防得住

声称调了工具？用 tool_call_required + require_referenced 核验签名留痕。声称改了文件？用 changed_files_verified 对基线指纹。「过程对终态错」？上 world_state 终态比对。

6 · fail→pass 闭环入库

samples 必须有一份挂掉的提交和一份通过的提交，replay.yaml 编排「失败→按反馈修正→通过」。这既验证题目可解，也验证反馈有引导力。

7 · validate 是底线不是终点

结构、引用、权重和、深度 lint 全过只说明「能跑」；题目质量还要靠 scripted 回放 + 真人 manual 各打一遍基线。

8 · 防刷分

反馈粒度按需收紧（summary 最严），cooldown_seconds 限制提交频率，max_attempts 卡轮次——防止把反馈当 oracle 穷举。

第 5 步 · 批量评估、可靠性口径与多 Agent 对比

单个 run 回答「这次做对了吗」；suite 回答「这个 Agent 在我的全部任务上稳定吗、比别的 Agent 强吗」。

suite 的三种用法

bash
# 1. CI 冒烟门禁（默认 scripted）：批跑全部任务的确定性回放，验证框架闭环未劣化
bin/agent-eval suite --tasks-root tasks --fail-on-not-passed

# 2. 真实 Agent 度量：批量过全部任务，--repeat 3 按 pass^3 口径判可靠性
bin/agent-eval suite --agent cli --cmd 'claude -p "$(cat {instructions})" --dangerously-skip-permissions' \
    --label my-agent --repeat 3

# 3. 多 Agent 并列对比：任务 × Agent 矩阵 + 按稳定通过数排序的排名面板
bin/agent-eval suite --agents-file agents.yaml --repeat 2

# 按分层批跑子集（任务 task.yaml 里的 tier: smoke/regression/security/domain）
bin/agent-eval suite --tier smoke --fail-on-not-passed

# 跨 run 趋势：汇总历次 report.json → 任务×Agent 的通过率/分数趋势（只读离线）
bin/agent-eval history --runs-root runs

pass^k：抓出「首跑过、复跑挂」的不稳定 Agent

借鉴 tau-bench：同一任务跑 k 次，k 次全过才算稳定通过。拖动滑块感受口径变化：

k = pass^1
agent-A（稳定）
agent-B（首跑过、复跑挂）

k=1 时两个 Agent 都「通过」；k≥2 时 agent-B 现形。报告（suite_report.md）里每个任务一行，稳定通过数决定排名。

成本与时延也在报告里

  • 每个 run 记录真实墙钟时延，套件报告聚合平均耗时。
  • Agent 在提交信封里自报 usage（model / token / cost）时，单 run 与对比面板会聚合出成本列——自报数据只做 ROI 参考，不参与评分，且以签名留痕事件记录。

接进 CI：四道闸模板

仓库自带 .github/workflows/ci.yml 与 bin/ci-smoke.sh，本地/CI 同口径：

  1. bin/gen-codemap.sh --check —— CODEMAP 漂移门禁：代码地图与源码不一致即失败（毫秒级）。
  2. mvn clean verify —— 159 项测试 + Checkstyle + JaCoCo 覆盖率下限 0.75。
  3. suite --fail-on-not-passed —— 任一任务的 fail→pass 回放闭环被破坏即退出码 3。
  4. redteam/run_all.sh —— 红队演练 fail-closed（Docker 就绪 19 项全 DEFENDED）：防线退化、观测偏离、报告缺失任一发生即失败。
连接真实系统工具默认走应答库回放（CI 永远确定性）。要连真实 API：在 task.yaml 给工具声明 http 后端（URL 静态白名单、凭证经 ${ENV:*} 从框架环境解析），用 AEL_TOOL_MODE=live 跑一次，框架把每次交换按应答库同格式存档到 <run>/tools/——复制进任务 hidden/tools/ 即完成「录制 → 回放」晋升。

常见问题 FAQ

Q1bin/agent-eval 报「未找到 CLI jar」？

先 mvn -q package。bin/agent-eval 只是一个定位 target/agent-eval-lite-*-cli.jar 的薄包装脚本，构建一次之后任意目录都能调。

Q2我的 Agent 某轮没写提交，会被判低分吗？

不会「误判」。该轮记为无提交，run 按剩余轮次/时间继续或结束，报告如实呈现。两类环境问题会被前移拦截而不是记 Agent 低分：http 服务不可达按评估基础设施故障处理；Docker 未安装/未启动直接退出码 1。

Q3怎么调试「为什么这条 check 挂了」？

三步：① 看 runs/<…>/judge/attempt_NNN.judge.json——完整判分含每条 check 的 earned/possible 与 private_notes；② 用 agent-eval judge --task … --submission … 离线复算，改规则后立刻验证；③ 建题阶段先跑 validate，深度 lint 能把断链/缺文件/越白名单等配置错误提前拦下。

Q4反馈会不会把答案泄露给 Agent？

机制上不会：反馈只含分数与失败检查的对外文案（feedback_fail），expected 值与 private_notes 永不回传。要更严可把 feedback.level 调成 summary（只给分），并用 cooldown_seconds 限制提交频率防穷举。注意：对外文案是任务作者写的，自检别把答案写进去。

Q5多次评估分数会不一样吗？

判分侧：14 种确定性 check 同输入必同分（这是可复现承诺，有指纹背书）；唯一例外 llm_rubric 会如实标注 deterministic=false 且权重被限制在 30% 以内。Agent 侧：模型本身有随机性，所以用 suite --repeat k 的 pass^k 口径度量稳定性。

Q6评不可信 / 强对抗 Agent 要注意什么？

务必 --sandbox docker：容器只挂 workspace/inbox/feedback/instructions，hidden 不在容器文件系统里，默认断网。镜像自备 Agent 运行环境（用工具还需 JRE）。非容器模式有两条如实登记的残余风险（外科式偷看、nonce 重放），只适合受控 / 半可信场景。

Q7任务想连真实系统（真实 API）怎么办？

在 task.yaml 给工具声明 backend: {type: http, url: …}（URL 静态白名单，凭证 ${ENV:KEY} 只在框架进程解析），跑一次 AEL_TOOL_MODE=live 录制真实应答，把 <run>/tools/*.recorded.yaml 拷进任务 hidden/tools/——之后 CI 全部走确定性回放。Agent 改不了外呼目标（入参只作请求负载）。

Q8跑了很多次，怎么看趋势和横向对比？

单次看 report.md；批量看 suite 的任务×Agent 矩阵与排名面板；历史趋势用 agent-eval history（只读聚合 runs/ 下历次 report.json，产出通过率与分数趋势）；要图形化看板则 export 推 OTLP 到 Phoenix / Langfuse。

常见坑清单（踩过一次就记住了）

建题时

  • task_id ≠ 目录名 → validate 直接拒
  • 维度权重之和 ≠ max_score → 加载 fail-fast
  • 把期望值写进 feedback_fail 文案或 work/ → 泄题（lint 管不了文案，靠自检与 canary）
  • llm 检查权重 >30% 或设 blocking → deep lint 拒
  • 录制的应答库忘拷进 hidden/tools/ → replay 模式该工具调用 fail-closed 失败

接入 / 运行时

  • 提交文件名必须 attempt_NNN.json 且与信封 attempt_id 一致，对不上判 invalid
  • 信封 summary 太短（<8 字符）→ Schema 校验失败

它针对的三个根本痛点

自然语言输出无法验收

Agent 说「我完成了」，但产出散落在对话里，没法机器判分。
→ 解法：结构化提交契约。唯一有效产出是写进 inbox 的 JSON，信封 + 分型双重 Schema 校验，聊天内容一概不计分。

Agent 自报不可信

说调了工具、说改了文件、说测试过了——都可能是编的。
→ 解法：过程留痕 + 交叉核验。工具调用经签名网关留痕，文件改动对指纹基线，测试在隐藏副本里真跑，终态与期望比对。

评测结果无法复现审计

「上周它能过」没法证明，评分标准也可能被摸透刷分。
→ 解法：隐藏判分 + 四指纹。规则与答案在 Agent 永不可见的 hidden/，评分携带引擎/规则/提交/工作区指纹，可随时离线复算。

设计红线（为什么它是「Lite」）

  • 零外部服务：不依赖数据库、消息队列、SaaS 平台；所有状态落文件系统，一个 run 目录就是完整档案。
  • 单 jar 交付：Java 17 + 4 个运行期依赖，mvn package 出一个 shaded jar，任何机器 java -jar 即用。
  • 目录即隔离：Work / Hidden / Judge 靠目录物理分区 + 指纹 + canary 探针守卫；对不可信 Agent 再叠加 Docker 容器强隔离。
  • 确定性优先：15 种判分 check 里 14 种完全确定性；唯一的 LLM 判分被限制在低权重主观维度且禁止一票否决。15 种类型的分类全景表见用户视角第 4 步。

第 4 步 · 接入路径与适用场景

评估者视角三步走，从零到接入 CI 大约半天；五类典型场景对号入座。

落地三步走

  1. 选题：直接用 5 个内置任务摸底（代码修复 / API payload / 文档分析 / 工具调用纪律 / 需求评审），或 task init 为自己的业务出题。任务是纯目录资产，可以在团队间像代码一样评审与共享。
  2. 接 Agent：四种适配（人工 / 回放 / 命令行 / HTTP 服务）覆盖绝大多数形态；不可信 Agent 叠加 --sandbox docker。接入成本 = 写一条命令模板或实现一个窄口径 HTTP 应答。
  3. 看结果、接 CI：单 run 看 report.md；批量看 suite 矩阵与排名；CI 用 --fail-on-not-passed 退出码 3 挡 PR；过程数据 export 进 Phoenix / Langfuse 看板。

五类典型场景

① 内部 Agent 能力度量

把自研/采购的 Agent 在同一套任务上打分，维度得分 + 轮次轨迹说明「差在哪」。人工基线（manual）与 Agent 分数直接可比。

② 回归门禁

Agent 升级了 prompt / 模型 / 工具链？suite 批跑全部任务，闭环劣化立刻退出码 3。评估自身也有红队回归守着，防线不静默退化。

③ 多 Agent 选型

agents.yaml 把候选 Agent（含服务型）拉到同一考场，任务 × Agent 矩阵 + pass^k 稳定性 + 耗时/成本聚合，选型证据一页出。

④ 安全与纪律评估

工具调用纪律（必须真调、禁止越权）、泄露探针、终态比对——评的不只是「会不会做」，还有「守不守规矩」。

⑤ 过程可观测

长跑任务开 auto-eval 后台采样，得到「分数随时间演进」的过程曲线；trace 导出 OTLP 后在看板上逐 span 复盘。

⑥ 评估资产沉淀

每道题自带 fail→pass 回放样例，任务库本身就是可持续回归的资产；录制→回放机制让真实系统交互也能沉淀为确定性用例。

ROI 相关的两个细节

  • 成本可见：Agent 自报 usage（token/成本）会聚合进对比面板，但明确「只做参考、不参与评分、留痕可审计」——避免把不可信数据混进结论。
  • 时延可比：逐 run 墙钟时延自动采集聚合，「谁又快又稳」一栏可见。

攻防矩阵（每张卡 = 一类已演练的攻击）

诚实声明：仍存的残余风险

非容器模式的「外科式偷看」

不开沙箱时 Agent 与框架同机同用户：只 sed 出隐藏答案数值、不整段转存也不触碰 canary，则无痕。对不可信 Agent 请用 --sandbox docker，容器隔离是根治手段（此项在红队基线中如实登记）。

非容器模式的 nonce 重放

command 检查的防短路 nonce 经命令行传给隐藏 harness，理论上进程可运行时读取自身命令行重放。挡得住「事先写死令牌」，挡不住「运行时动态重放」——同样由容器隔离根治。

run 结束后的产物篡改

签名密钥在 run 收尾（Agent 已停止）才落盘供离线复核；对已完成 run 的离线篡改属另一层威胁，靠目录指纹 + 从干净任务目录复算发现。

Docker 沙箱的边界

隔离强度取决于 Docker 本身（默认 root-in-container、共享内核）；镜像由评估方提供需自行确保可信；默认断网下工具网关不可达，需工具的任务用 bridge 回连。

结论口径默认（进程内硬化）模式：能挡住常见作弊手法，适合受控的真实业务评估（内部度量、CI 门禁、多轮回归）。启用 --sandbox docker 后补齐「外科式偷看」与「运行时 nonce 重放」，可面向不可信 / 强对抗 Agent。

五条设计原则（读代码前先装进脑子）

1 · 文件即状态

没有数据库。run 目录自包含：resume、离线重评、报告重建都只凭目录还原上下文。好处是评估产物天然可归档、可 diff、可审计。

2 · fail-closed

加载校验不过就 fail-fast；工具白名单外一律拒绝；应答库缺失调用失败而不是放行；LLM 判分连不上按评审设施故障上抛，绝不静默打分。

3 · 确定性优先

判分默认全确定性（同输入必同分）；唯一非确定性的 llm_rubric 被 deep lint 强制「权重 ≤30%、禁止 blocking」，并在结果里如实标注。

4 · 不可变值对象

TaskSpec、AttemptInput/Outcome、JudgeResult 等全是 record；Judge 只依赖 JudgeInput 的只读视图，禁止读取 runs 之外的可变状态。

5 · Agent 是黑盒，信任来自核验

框架从不解析 Agent 的思考过程；一切结论都建立在可核验的物证上——签名留痕、指纹比对、隐藏副本里真跑。

6 · 评审不污染现场

command 型检查在 workspace 临时副本执行（ephemeral）；auto-eval 采样产物隔离在 judge/auto/。Agent 永远看不到评审的中间产物。

RunManager.execute 的九个动作

  1. 装载任务 —— TaskSpecLoader.load() 读 task.yaml，fail-fast 校验（task_id 与目录一致、维度权重和 = 满分、引用文件存在），产出不可变 TaskSpec。
  2. 准备现场 —— 生成 run_id 与目录骨架；WorkspaceManager 把 work/ 复制为私有 workspace，记录文件级 SHA-256 基线，并计算 hidden/ 目录树指纹。
  3. 渲染说明 —— InstructionsRenderer 由 TaskSpec 生成 instructions.md：题面 + 可见清单 + 提交契约 + 反馈位置（不含任何 hidden 信息）。
  4. 建立可信通道 —— TraceSecret.obtain() 生成每 run 私密签名密钥（只存内存）；ToolGatewayServer.start() 在 loopback 临时端口拉起常驻工具网关，凭 token 访问。
  5. attempt 循环 —— 对每轮调用 adapter.runAttempt(AttemptInput)（阻塞至 Agent 结束或超时），期间 Agent 读说明、干活、可经网关调工具、把 JSON 写进 inbox。
  6. 收提交 —— SubmissionManager 做信封 + 分型双重 Schema 校验与 task/attempt 绑定校验；不合格记 submission_invalid，反馈 schema 错误清单，消耗一轮但不进评分。
  7. 判分与反馈 —— JudgeRunner 组装只读 JudgeInput 交给 RulesJudge（可叠加 ScriptJudge）；完整结果落 judge/，FeedbackPolicy 按任务配置裁剪出受控反馈写 feedback/。达线提前结束；Agent 自称完成但未达线且有余量 → 记 stop_hook_triggered 携反馈续轮。
  8. 逐轮快照 —— RunStateStore 原子落盘 run_state.json；--resume 校验 hidden 指纹后从下一轮继续，被篡改则 INTEGRITY_BROKEN 熔断。
  9. 终局 —— BestAttemptSelector 按 selection 策略（best_score / first_pass / last）选最佳轮；ReportGenerator 产出 report.json/md；收尾时（Agent 已停止）TraceSecret.save() 才把签名密钥落盘供离线复核与 resume。

适配器 SPI：接入面有多小

看完这个接口你就明白为什么「任何东西」都能接进来——适配器唯一的义务是：在超时预算内驱动 Agent，让提交出现在 inbox。

/** Agent 适配器 SPI：把「任何能产出提交文件的东西」接入评估循环。 */
public interface AgentAdapter {
    String name();
    /** 执行一轮 attempt：驱动 Agent 工作直至其产出提交或放弃。 */
    AttemptOutcome runAttempt(AttemptInput input);
}

/** 一轮的输入：任务上下文 / attempt id / 说明文件 / 上轮反馈 / 超时预算 / 工具通道 */
public record AttemptInput(TaskContext context, String attemptId, int attemptNumber,
        Path instructionsFile, Path previousFeedbackFile, Duration timeout, ToolAccess toolAccess) { }

/** 一轮的结果：提交文件（可为 null）/ 是否自称完成 / 退出码 / 日志 / 是否已无后续输入 */

② trace 防伪造：HMAC 签名链路

  1. TraceSecret.obtain() 每 run 生成私密密钥——只存框架内存，Agent 执行期间不落盘（这是安全不变量）。
  2. TraceLogger 写每条事件前由 TraceSigner.sign() 计算 HMAC-SHA256，写入 _sig 字段，append-only。
  3. 判分侧只统计 TraceSigner.verify() 通过的 tool_call 事件——Agent 直接往 trace.jsonl 手写伪造事件（无合法签名）被静默丢弃。
  4. run 收尾（Agent 已停止）密钥才落盘，供离线复核与 resume 续签。

③ 工具网关：框架进程内的常驻服务

  • ToolGatewayServer 监听 loopback 临时端口，token 鉴权；Agent 侧 agent-eval tool call（ToolGatewayClient）经它调用——网关是「框架的手」，不是 Agent 派生的子进程，所以留痕可信。
  • ToolGateway 按任务 allowed_tools fail-closed 拦截，未授权工具记 error=tool_not_allowed 且不产生成功调用记录。
  • 应答两条通道：replay（默认，hidden/tools 应答库，零外呼）与 live（ToolHttpBackend 按 task.yaml 静态声明的 URL 真实外呼，同格式存档供晋升回放）。URL/method/headers 全静态声明，Agent 入参只作请求负载，改不了外呼目标。
  • Docker 沙箱 + bridge 网络时，容器内经 host.docker.internal 回连宿主网关，token 仍作纵深防御。

第 5 步 · 二次开发指南

四条最常见的扩展路线，按改动面从小到大排序。共同纪律：新能力必须带测试，判分相关的还要考虑红队回归。

路线 A新增一个任务（不写 Java，最常见）

这就是用户视角第 4 步的建题五步法：task init 脚手架 → 改 task.yaml 与 work/ → 写 hidden 规则 → validate → samples 闭环。任务是纯目录资产，合入后 suite 批跑自动把它纳入 CI 回归。

约定：任务 ID kebab-case 加数字后缀；YAML 字段 snake_case；评审专用数据只进 hidden/，别把 expected 复制到 work/、公开文档或样例。

路线 B新增一种判分 check
  1. 在 RulesJudge 的 switch 加一个 case，实现私有方法：读 def（规则里的参数）与 JudgeInput（只读视图），返回 CheckOutcome。禁止读取 runs 之外的可变状态，保持确定性。
  2. 若 check 引用 hidden 文件或工具，在 ValidateCommand 的深度 lint 里加对应的前移拦截（断链/缺失/越白名单在 validate 时报错，而不是 run 时炸）。
  3. 在 src/test/java/com/agenteval/judge/ 加单测（正例/反例/边界），必要时给内置任务或 testsupport 夹具加用例。
  4. 如果这个 check 承担防作弊职责，在 redteam/ 加一个攻击演练目录并登记进 run_all.sh，把防线钉进 CI。
路线 C新增一种 Agent 适配器
  1. 实现 AgentAdapter：在 runAttempt 里驱动你的 Agent（进程/RPC/消息队列都行），遵守 input.timeout()，结束时返回 AttemptOutcome（提交文件路径或 null）。参考 HttpAgentAdapter（约 200 行）最容易上手。
  2. 在 RunCommand.buildAdapter() 与 SuiteCommand（含 agents.yaml 解析）接线新的 --agent 取值与参数。
  3. 把「Agent 必须知道的事」都从 AttemptInput 取：instructions 路径、inbox、attempt_id、上轮反馈、工具通道 ToolAccess——不要私自读 run 目录其他位置。
  4. 加集成测试（integration 包）：用假 Agent 跑通 fail→pass 闭环；外部依赖不可用时用 assumption 自动跳过（参考 Docker 用例的做法）。
路线 D新增一个 CLI 子命令
  1. 在 com.agenteval.cli 建 picocli @Command 类（参考 ExportCommand，最小）。
  2. 注册进 Main 的 subcommands 列表。
  3. 遵守退出码约定：0 正常 / 1 参数输入错误 / 2 框架故障 / 3 门禁未过；错误信息走 stderr。
  4. 只读命令保持幂等（对齐 report 的「从工件重建、纯读」姿态）。

代码风格速记

  • 四空格缩进；类/record 用 PascalCase，方法字段 camelCase；值对象偏向不可变 record 或 final 类。
  • 公开 API 按现有风格写中文 Javadoc，核心类型保留 @author 与 @since；Checkstyle 在 CI 强制。
  • 单测 *Test.java 与被测包同路径；端到端进 integration 包。

第 6 步 · 调试、测试与质量门禁

评估框架自己也要被评估。这一步讲：怎么排障、框架用什么守住自己的质量。

排障工具箱

分数可疑 → 离线复算

agent-eval judge --task … --submission … 对任意历史提交重评；完整判分细节（含 private_notes、每条 check 的 earned/possible）在 judge/attempt_NNN.judge.json。

报告丢了 → 从工件重建

agent-eval report --run runs/xxx/run_yyy 纯读幂等地重建 report.json/md——报告永远是 trace + judge 工件的投影，不存独立状态。

进程被杀 → 断点续跑

run --resume <run 目录> --agent … 从 run_state.json 的下一轮继续；恢复前校验 hidden 指纹，被篡改则 INTEGRITY_BROKEN 熔断（退出码 2）。

过程看不清 → trace / 看板

直接读 traces/trace.jsonl（每行一个签名事件），或 export 成 OTLP 推到 Phoenix / Langfuse 逐 span 复盘；auto-eval 采样事件里有分数轨迹。

框架自身的三层质量网

  1. 单元 + 端到端测试（159 项） —— judge 引擎逐 check 正反例、submission 校验、trace 签名、report、resume，加上 5 个内置任务的 fail→pass 回放端到端；mvn clean verify 同时跑 Checkstyle 与 JaCoCo 指令覆盖率 ≥ 0.75 下限。
  2. 任务集回放门禁 —— suite --fail-on-not-passed：任何改动破坏了任一任务的确定性闭环，CI 直接红。
  3. 红队回归（fail-closed） —— redteam/run_all.sh 演练 16 类攻击（偷看 hidden、伪造 trace、command 短路、越权工具、canary 外泄、终态造假…）。判定逻辑抽为 gate_lib.sh 纯函数并由 test_gate.sh 正/负向自测钉死：VULNERABLE 超出登记基线、观测偏离预期、报告缺失——任一发生即失败，产出结构化工件 redteam_report.json。
红队基线的自适应Docker 就绪时红队 A 在容器内演练，19 项全 DEFENDED、登记基线 0；无 Docker 则回退非容器路径（17 项）并如实保留 1 项 VULNERABLE 登记（外科式偷看）。CI 另设强制 Docker job（RT_REQUIRE_DOCKER=1 且 RT_ALLOWED_VULN=0，镜像按 digest 固定），专门防「隔离能力静默降级」。
