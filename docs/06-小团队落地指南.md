# 小团队落地指南

本指南面向 2-6 人、1-3 个 Agent 的内部团队。目标不是先做平台，而是先把“自己的 Agent 能不能稳定完成关键任务”测清楚。

## 推荐路线

优先级从高到低：

1. 私有 evalset 骨架：`bin/agent-eval evalset init --id <set-id>`
2. 10 个 smoke 任务：每次改 Agent / prompt 必跑。
3. 20-50 个 regression 任务：合并前或发版前跑。
4. 固定 `agents.yaml` 和 `scripts/run-agent.sh`：统一模型、鉴权、环境变量、提交路径。
5. 使用 `suite_report.md` 看未稳定通过任务、失败规则热点、pass^k 可靠性。
6. 用 `ael-review-task-quality` 审查任务能否进入门禁。

暂时不要优先做：Web 平台、多租户、权限、分布式调度、复杂 dashboard、大量假业务任务。

## AI 辅助入口

如果团队使用 Codex、Claude Code 或其他支持项目级 skills 的 AI 编码助手，建任务时优先让 AI 走仓库内置 skills，而不是从空目录手写：

| 场景 | 让 AI 使用 |
| --- | --- |
| 从零建小团队测评集 | `ael-build-evalset` + `bin/agent-eval evalset init --id <set>` |
| 没有具体业务、先补骨架 | `evalsets/_template/` 或 `evalset init` |
| 在私有集里新增任务 | `ael-new-task` + `bin/agent-eval task init --tasks-root evalsets/<set>/tasks` |
| 判断任务能否进门禁 | `ael-review-task-quality` + [07-任务质量清单.md](07-任务质量清单.md) |
| 改完后验证 | `ael-verify` |
| 跑完后分析失败 | `ael-analyze-results` |

推荐对 AI 这样说：

```text
用 ael-build-evalset 帮我在 evalsets/my-agent 建小团队测评集；
没有真实业务题时先用 evalset init 和 evalsets/_template；
新增任务用 ael-new-task，进 smoke/regression 前用 ael-review-task-quality 审查。
```

## 初始化

```bash
bin/agent-eval evalset init --id my-agent
```

生成：

```text
evalsets/my-agent/
├── README.md
├── agents.yaml
├── scripts/run-agent.sh
└── tasks/.gitkeep
```

也可以直接复制 `evalsets/_template/`。

## 建第一个任务

```bash
bin/agent-eval task init --id first-task-001 --tasks-root evalsets/my-agent/tasks

bin/agent-eval validate --task evalsets/my-agent/tasks/first-task-001

bin/agent-eval run --task evalsets/my-agent/tasks/first-task-001 \
    --agent scripted \
    --script evalsets/my-agent/tasks/first-task-001/samples/replay.yaml \
    --runs-root evalsets/my-agent/runs
```

每个任务进门禁前必须满足：

- `task.yaml` 清楚描述能力目标和提交要求。
- `work/` 材料足够 Agent 独立作答。
- `hidden/` 只放 expected、judge 规则和 mock 响应。
- `samples/attempt-pass.json` 能过。
- `samples/attempt-fail.json` 犯真实 Agent 可能犯的错。
- `samples/replay.yaml` 能跑出 fail -> pass 闭环。
- `feedback_fail` 不泄露 hidden 期望值。

完整清单见 [07-任务质量清单.md](07-任务质量清单.md)。

## 接入 Agent

把团队自己的 Agent 封装到：

```text
evalsets/my-agent/scripts/run-agent.sh
```

Agent 只需要遵守一个契约：把 JSON 提交写入 `$AEL_INBOX/$AEL_ATTEMPT_ID.json`。

单任务冒烟：

```bash
bin/agent-eval run --task evalsets/my-agent/tasks/first-task-001 \
    --agent cli \
    --cmd 'bash "$AEL_RUN_DIR/../../../scripts/run-agent.sh" current' \
    --runs-root evalsets/my-agent/runs
```

多 Agent 横评：

```bash
bin/agent-eval suite --tasks-root evalsets/my-agent/tasks \
    --runs-root evalsets/my-agent/runs \
    --agents-file evalsets/my-agent/agents.yaml \
    --repeat 3
```

## 看报告

`suite_report.md` 的阅读顺序：

1. `总体结论`：是否全部稳定通过。
2. `小团队操作摘要`：先看未稳定通过任务、故障任务、flaky 任务和失败规则热点。
3. `逐任务明细`：看具体任务得分、pass^k 和耗时。
4. 单任务下钻：打开对应 run 的 `report/report.md`、`feedback/`、`agent-logs/`。

常见定性：

| 现象 | 优先判断 |
| --- | --- |
| 所有 Agent 都挂同一个任务 | 先检查任务材料、judge 规则或环境 |
| 只有一个 Agent 挂 | 多半是 Agent 能力或接入问题 |
| pass@1 过但 pass^k 挂 | Agent 不稳定，不能当稳定能力 |
| score 为 null | 没有有效提交，先查接入契约 |
| status 为 ERROR | 框架/环境故障，不应算 Agent 低分 |

## 推荐门禁

小团队第一阶段：

```bash
# 快速回归
bin/agent-eval suite --tasks-root evalsets/my-agent/tasks \
    --runs-root evalsets/my-agent/runs \
    --tier smoke \
    --agents-file evalsets/my-agent/agents.yaml \
    --repeat 2

# 发版前
bin/agent-eval suite --tasks-root evalsets/my-agent/tasks \
    --runs-root evalsets/my-agent/runs \
    --agents-file evalsets/my-agent/agents.yaml \
    --repeat 3
```

建议达标线：

- smoke：核心 Agent 必须 100% pass^k。
- regression：按业务风险设通过率，先从 80%-90% 起。
- security：失败需要人工确认后再放行。

## 配套 skills

- `ael-build-evalset`：把业务场景变成 evalset。
- `ael-new-task`：新增或修改任务。
- `ael-review-task-quality`：任务进入 smoke/regression 前做质量审查。
- `ael-analyze-results`：评测跑完后做失败归因。
- `ael-verify`：提交前选择最小验证门禁。
