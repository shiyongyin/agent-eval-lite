## 2. 目录结构摘要

严格六层架构（ArchUnit 11 条规则强制），依赖方向：

```text
interfaces  -> application            （REST/SSE 控制器 → 用例；端口定义在消费侧）
ai          -> application ports + domain   （AgentScope 适配环，与 infrastructure 同构）
infrastructure -> domain/application/ai ports
```

```text
src/main/java/com/company/agentapp/
  bootstrap/        入口 AgentAppApplication + config/*Properties（SSOT 配置面，禁散落 @Value）
  interfaces/       rest（Chat/Usage/Conversation/ToolApproval Controller）、sse、security（TenantWebFilter）
  application/      chat / quota / approval / session 用例 + agentops（eval/feedback/versioning 蓝图区）
  ai/               agent（AgentFactory/ToolkitFactory/PromptProvider）、hooks、safety、rag、
                    runtime（AgentSessionManager 唯一执行入口）、trace（RunTrace*）、
                    evaluation（Evaluator/Composite/Rule/Judge/EvalRunner）、tools、replay、skill
  domain/           order、knowledge（纯 Java，无 Spring/AgentScope）
  infrastructure/   agentscope（Model/Session/Mcp Config、RedisAgentStateStore）、persistence（InMemory*）、sandbox
  shared/           error / security / util
src/main/resources/ application.yml、application-prod.yml
src/test/java/      与主代码同构 + architecture/（ArchitectureRulesTest、DeadCodeGuardTest）+ integration/
docs/               adr/（17 篇 ADR）、not-wired-ledger.md、design-backlog.md、红队报告、site/
```

- 入口文件：`bootstrap/AgentAppApplication.java`；核心 API `POST /api/v1/chat/stream`（SSE）。
- 配置：`application.yml`（`agentapp.*` 强类型绑定 `bootstrap/config/*Properties`）。
- 无 scripts/、tools/、examples/ 目录；docs/ 齐备。

## 5. 评估框架放置位置（结论）

**推荐：在 `/Users/mac/work/` 下新建独立项目 `agent-eval-lite`（独立 git 仓库、独立 Maven 工程）。**

不放进 agentScopeScaffold 的四条硬理由：

1. **ADR-0017 明文决策**「不自建 AgentOps 平台，评测走标准出口 + 外接平台」——在仓库内自建评估框架直接违背其已接受的 ADR。
2. **DeadCodeGuardTest / NOT-WIRED 台账纪律**：评估框架对 Spring 应用而言是「零运行时消费方」的旁路系统，入仓即触发死代码守护或迫使台账膨胀。
3. **生命周期不同**：评估框架的被测对象是任意 Agent（Claude Code / Codex / Cursor CLI / 人工 / HTTP 服务），与该 Spring 应用的发布节奏、依赖基线（AgentScope SNAPSHOT pin）完全无关。
4. **角色反转更干净**：agentScopeScaffold 本身应是**被评估对象之一**（Phase 2 经 HTTP adapter 接它的 `/api/v1/chat/stream`）——裁判与选手不同仓，正是 Work/Judge 隔离思想在仓库粒度的体现。

技术栈对齐：新项目沿用 Java 17 + Maven + Jackson + JUnit 5 + SLF4J（与现有工具链、团队技能一致），**不引入 Spring**（CLI harness 不需要容器），CLI 用 picocli，schema 校验用 networknt json-schema-validator。

## 6. 初步风险点

| # | 风险 | 应对 |
|---|---|---|
| R1 | Phase 1 本地目录隔离对「拥有任意 shell 的 coding agent」只是约定 + 审计，非对抗性强隔离 | 明确标注适用边界（合作式评测）；对抗性评测等 Phase 4 Docker；judge 侧校验 workspace 越界痕迹 |
| R2 | LLM judge 可复现性弱 | 确定性规则优先；LLM judge 仅限主观维度，rubric + 温度 0 + 版本/模型记录，Phase 2+ 才接 |
| R3 | trace 含敏感原文 | 评测环境默认只用 mock/合成数据；借鉴 ADR-0012，raw 内容截断 + 标注 |
| R4 | CLI coding agent 输出格式漂移 | 提交走**文件契约**（写 submission.json 到指定 inbox），不解析 stdout |
| R5 | 评测成本失控 | max_attempts / timeout_minutes / cooldown_seconds 为 TaskSpec 一等公民 |
| R6 | 隐藏规则经反馈泄露（EdgeBench 论文附录 C 的「feedback as oracle」攻击） | 反馈粒度可配（feedback 段），expected 数据不回显，命中值不回传 |
